# 프로젝트 레벨 및 경쟁력 분석

각 프로젝트의 **구현 난이도**, **필요 개발자 레벨**, **예상 기간**, **포트폴리오 경쟁력**을 정리한다.
평가는 **현재 구현 상태**와 **설계 범위까지 모두 구현되었다고 가정했을 때**를 분리해서 쓴다.

---

## PART 1. 전체 개요 및 평가 기준

### 1. 프로젝트 설계 문서 위치

각 프로젝트에 대한 **가장 정확한 정보**는 해당 프로젝트의 `design/` 폴더에 있다.

* **arena/design/**: MVP 1.0–1.3 (게임 서버, 전투, 매치메이킹, 통계)
* **backend/node.js/design/**: N2.0–N2.6 (NestJS 패턴 훈련)
* **backend/spring/design/**: Milestone 2.0–2.6 (Spring Boot 패턴 훈련)
* **e-commerce/design/**: Stage 0–4 (전자상거래 플랫폼)
* **framework/design/**: Milestone 1.1–1.4.5 (프레임워크 기초)
* **game-server/design/**: Lab 1.1–1.4, netcode-core 1.0–1.3
* **react/design/**: Milestone 1–6 (React 패턴 훈련)
* **video-editor/design/**: Phase 0.0–3.0 (동영상 편집기)

> 이 문서와 design/ 내용이 다르면 **design/** 기준이 맞다고 본다.

---

### 2. 요약 비교표

| 프로젝트                | 필요 레벨 (주요 범위 기준) | 구현 기간(예상) | 신입 경쟁력     | 이직 경쟁력     | 주 타겟 포지션    |
| ------------------- | ---------------- | --------- | ---------- | ---------- | ----------- |
| **arena**           | 시니어              | 6–8주      | ⭐⭐⭐⭐⭐      | ⭐⭐⭐⭐⭐      | 게임 서버 개발자   |
| **video-editor**    | 시니어              | 10–12주    | ⭐⭐⭐⭐⭐      | ⭐⭐⭐⭐⭐      | 웹 앱 / Vrew  |
| **game-server**     | 미들~시니어           | 6–8주      | ⭐⭐⭐⭐       | ⭐⭐⭐⭐       | 게임 서버 개발자   |
| **framework**       | 시니어              | 3–4주      | ⭐⭐⭐⭐       | ⭐⭐⭐⭐       | 백엔드 전반      |
| **e-commerce**      | 주니어~시니어(단계별 상이)  | 18–36주    | ⭐⭐⭐⭐~⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐~⭐⭐⭐⭐⭐ | 백엔드 개발자     |
| **backend/node.js** | 주니어~미들           | 5–7주      | ⭐⭐⭐        | ⭐⭐⭐        | Node.js 백엔드 |
| **backend/spring**  | 주니어~미들           | 5–7주      | ⭐⭐⭐(예상)    | ⭐⭐⭐(예상)    | Spring 백엔드  |
| **react**           | 주니어~미들           | 4–6주      | ⭐⭐⭐(예상)    | ⭐⭐⭐(예상)    | 프론트엔드 / 풀스택 |

※ **별점/레벨 기준**

* ⭐⭐⭐⭐⭐

  * 대부분의 개발자가 시도하지 않는 수준의 기술 혹은 조합
  * 저수준(C++/네트워크/FFmpeg 등) + 프로덕션 품질까지 갖춘 경우
* ⭐⭐⭐⭐

  * 실무 상위권 수준의 기술 스택, 관측성, 확장성까지 포함
  * 경쟁 포트폴리오 대비 분명한 강점은 있으나, “보기만 해도 압도적인” 레벨까지는 아님
* ⭐⭐⭐

  * 시장에서 흔히 보이는 스택/패턴
  * “될까?” 수준이 아니라 “기본은 된다” 정도의 포트폴리오

※ **평가 기준이 어떤 상태를 가정하는지**

* **현재 구현 기준으로 평가한 것**

  * arena, video-editor, framework, backend/node.js
* **“해당 단계까지 전부 구현되었다고 가정한 예상치”가 섞인 것**

  * game-server: Track A+B를 모두 완료했다고 가정한 난이도/경쟁력
  * e-commerce: Core / Scale / Cloud 각 체크포인트 완료 가정 시 신입/이직 경쟁력 별도 표기
  * backend/spring, react: 현재는 설계만 존재 → “완료 시 예상”으로만 별점 표기, **현재 상태는 별점 없음(–)**

아래 각 프로젝트 섹션에서 **“현재 구현 기준” / “완료 시 예상”**을 명시해서 다시 정리한다.

## PART 2. 지원 전략 전체 설계

### 1. 한국 회사 티어 분류

**Tier 1 – 대형 테크 / 대형 게임사 (대략 연 7천+ 가정)**

* 예: Naver, Coupang, 토스, Nexon, Krafton, Kakao Games, Voyager X 등
* 채용 프로세스 길고(3–5 라운드), CS/아키텍처/코딩테스트 비중 큼
* 포트폴리오만으로 합/불이 결정되진 않지만, **“걸러지는 쪽인지, 볼만한 쪽인지”**는 영향을 줌

**Tier 2 – 중견 게임사·성장 스타트업 (5–7천)**

* 예: Netmarble, NCsoft, 펄어비스, 우아한형제들, 마켓컬리 등
* 실무 중심 면접(2–3 라운드) 비중이 더 큼
* 포트폴리오는 “실제로 써먹을 수 있는 사람인지” 판단 재료

**Tier 3 – 중소/초기 스타트업 (4–5천)**

* 채용 절차 단순, 인원 적음
* 포트폴리오보다 **당장 업무 투입 가능성**과 “같이 일할 수 있느냐”를 더 봄

이 문서의 전략은 **“포트폴리오로 최소한 서류/면접 테이블에는 올라간다”**는 정도를 목표로 한다.
“이걸 하면 합격 확정” 같은 식으로는 보지 않는다.

---

### 2. 포지션별 조합 전략 (요약)

#### 게임 서버 개발자 (Tier 1 목표)

* **조합 추천**: arena + game-server (+ 선택적으로 framework)
* **의도**

  * arena: “실제 돌아가는 실시간 1v1 게임 서버”
  * game-server: 엔진/넷코드 레벨의 기초·심화
  * framework: 프레임워크 내부 이해(선택)

#### 웹 앱 / Vrew 타겟

* **조합 추천**: video-editor 단독으로도 충분히 강함
* 필요 시 backend/node.js, react를 붙여서 “웹 전반 경험” 보완 가능

#### 백엔드(Spring/Node)

* **조합 추천**

  * Spring: framework → backend/spring
  * Node: framework → backend/node.js
* **의도**

  * “프레임워크 안쪽이 뭔지 알고 쓰는 사람”이라는 메시지 전달

#### 풀스택

* **조합 추천**: backend/node.js 또는 backend/spring + react (+ framework)
* 스타트업/소규모 서비스 쪽에서 풀스택 포지션 지원 가능

자세한 내용은 맨 아래 **시간 제약별 전략**과 각 프로젝트 섹션의 “지원 전략”에서 다시 다룬다.

---

### 3. 시간 제약별 전략

#### 4주 이내

* **현실적인 선택**: framework 단독
* 목표: “프레임워크 내부 동작 이해하는 백엔드 주니어”
* Tier 1보다는 Tier 2–3 백엔드 주니어용에 가깝다.

#### 8주 전후

* **백엔드 노선**

  * framework + backend/node.js 또는 framework + backend/spring
* **게임 서버 노선(압축)**

  * game-server의 Track A만이라도 완성해서 “C++ 네트워크 기초”를 보여주는 방향

#### 10–12주

* **게임 서버 강하게 노릴 때**: arena
* **Vrew / 웹+네이티브 강하게 노릴 때**: video-editor

둘 다 “일반 포트폴리오와 급이 다른” 인상까지는 가능하다.
(합격을 보장하지는 않지만, **포트폴리오 경쟁력 자체는 상위권**으로 봐도 된다.)

#### 3개월 이상

* e-commerce Core → Scale → Cloud 순서로 확장
* Tier 1 전자상거래/결제 도메인 백엔드 포지션까지 노릴 수 있는 구조

## PART 3. 프로젝트별 상세 분석 & 지원 전략

---

### 1. game-server (C++ 게임 서버 기초부터 프로덕션까지)

#### 📋 프로젝트 개요

* **Track A – gameserver-fundamentals**

  * TCP / WebSocket 기반 서버, 기본 매치/로직
* **Track B – netcode-core**

  * UDP + Protobuf 기반 넷코드 엔진
  * 상태 동기화, 매치 메이킹, 모니터링

#### 👨‍💻 필요한 개발자 수준

* Track A: 주니어~미들

  * C++ 기본기 + 네트워크 기초
* Track B: 미들~시니어

  * 비동기 IO, 패킷 포맷, 성능·안정성 고려 필요

#### ⏱️ 구현 기간(설계 기준)

* Track A: 약 2주
* Track B: 4–6주
* 합산 6–8주

#### 🎯 포트폴리오 경쟁력 (완료 시 기준)

* **신입 기준: ⭐⭐⭐⭐**

  * 일반 Unity/엔진 베이스 프로젝트에 비해 확실히 깊이 있음
  * 다만 arena처럼 “완성된 게임”까지는 아니어서 임팩트는 약간 덜하다.
* **이직 기준: ⭐⭐⭐⭐**

  * 게임 서버/네트워크 포지션에 충분히 쓸 수 있는 깊이
  * 단독으로 Tier 1을 “뚫는다”기보다는 arena와 조합했을 때 더 설득력 있음

#### 💡 추천 대상

* C++ 게임 서버 쪽으로 갈 생각이 확실한 경우
* arena 전에 “기초를 밟고 가고 싶다”일 때

#### 🎯 지원 전략 (요약)

* **Tier 2 게임사**(Netmarble, Smilegate 등) 쪽에서

  * “교육용 Track A + 심화 Track B” 구조를 그대로 설명하면 이해하기 좋다.
* **arena와 같이 가져갈 때**

  * game-server: “엔진/넷코드 레벨에서 다뤘다”
  * arena: “실제 게임으로 완성해 봤다”
  * 이런 식으로 역할을 분리해서 어필하는 게 좋다.

### 2. arena (실시간 1v1 대결 게임 서버)

#### 📋 프로젝트 개요

* 60 TPS WebSocket 실시간 게임 서버
* ELO 매치메이킹, 전투/랭킹 시스템
* PostgreSQL, Redis, Prometheus 연동

#### 👨‍💻 필요한 개발자 수준

* C++17 모던 문법에 익숙한 시니어급 또는 그에 준하는 수준
* 비동기 IO, 게임 루프, 상태 동기화 개념 필요

#### ⏱️ 구현 기간(현재 설계·구현 기준)

* MVP 1.0–1.3 합산 6–8주 정도 작업량

#### 🎯 포트폴리오 경쟁력 (현재 구현 기준)

* **신입 기준: ⭐⭐⭐⭐⭐**

  * 흔히 나오는 “토이 WebSocket 서버” 레벨이 아니라,
    실제로 1v1 게임이 가능하고, 60TPS + 모니터링까지 들어간 수준.
* **이직 기준: ⭐⭐⭐⭐⭐**

  * C++ 게임 서버 포지션에서 이야기할 거리 충분하다.
  * 단, 실제 회사의 대규모 상용 서버와는 스케일 차이가 있다는 점은 면접에서 같이 언급하는 편이 안전하다.

#### 💡 추천 대상

* 한국 게임사(특히 서버) 노릴 사람
* “C++로 진짜 게임 서버 짰다”라고 말하고 싶은 사람

#### 🎯 지원 전략

* **Tier 1 게임사(Nexon, Krafton, Kakao Games)**

  * 키워드:

    * “from scratch C++ 멀티플레이어 서버”
    * “60 TPS 고정 틱, p99 틱 분산 < 1ms”
    * “WebSocket + ELO 매치 메이킹”
  * 면접에서는 “스케일링은 어떻게 할 생각이냐” 질문이 나올 가능성이 높으므로,
    현재는 단일 서버 기준이고, 샤딩/룸 서버/매치 서버 분리 계획 정도를 설계 레벨에서 말할 수 있어야 한다.

### 3. video-editor (VrewCraft – 웹 기반 동영상 편집기)

#### 📋 프로젝트 개요

* React 18 + Node.js 20 + TypeScript 5
* C++ N-API + FFmpeg C API 직접 사용
* 썸네일 추출, 메타데이터 분석, 모니터링

#### 👨‍💻 필요한 개발자 수준

* 프론트엔드(React)와 백엔드(Node)를 둘 다 다룰 수 있어야 하고,
* C++로 FFI(네이티브 애드온)까지 들어가는, **복합 스킬**이 요구된다.

#### ⏱️ 구현 기간(현재 설계·구현 기준)

* Phase 0–3 합산 10–12주 정도 작업량
* 현재 Phase 3까지 완료된 상태라고 보는 설계

#### 🎯 포트폴리오 경쟁력 (현재 구현 기준)

* **신입 기준: ⭐⭐⭐⭐⭐**

  * 웹만 하는 사람과는 결이 다르다.
  * FFmpeg C API를 직접 다루는 것 자체가 흔하지 않다.
* **이직 기준: ⭐⭐⭐⭐⭐**

  * Voyager X(Vrew)처럼 실제로 영상 편집/처리를 하는 회사에 쓸 수 있는 수준.
  * 단, 실제 상용 서비스의 규모(트래픽, 장애 케이스 등)는 별도로 공부해야 한다.

#### 💡 추천 대상

* Vrew / 동영상 편집 도메인을 겨냥하는 경우
* “웹 + 네이티브 + 성능 최적화”라는 스토리를 만들고 싶은 경우

#### 🎯 지원 전략

* **Voyager X(Vrew)**

  * 채용 공고의 기술 키워드(React, Node.js, TypeScript, FFmpeg 등)와
    video-editor의 구성 요소를 1:1로 매핑해서 보여주면 설득력이 높다.
* **그 외 웹 회사(우아한형제들, 토스 등)**

  * “보통 웹 개발자가 잘 안 건드리는 영역(FFmpeg, C++)을 건드려 봤다”는 점을
    “문제 생겼을 때 한 단계 더 내려가서 파볼 수 있는 사람”이라는 메시지로 사용하면 된다.

### 4. e-commerce (Sagaline – 전자상거래 플랫폼)

#### 📋 프로젝트 개요

* Spring Boot 3.2 + Java 17
* PostgreSQL, Redis, Kafka, Elasticsearch
* 모놀리스 → 확장/관측성 → 클라우드/마이크로서비스로 단계별 설계

#### 👨‍💻 필요한 개발자 수준

* **Core(Stage 1–4)**: 주니어~미들
* **Scale(Stage 5)**: 미들
* **Cloud(Stage 6–9)**: 미들~시니어

#### ⏱️ 구현 기간(설계 기준)

* Core: 18주
* Scale: 추가 7주 (합 25주)
* Cloud: 추가 11주 (합 36주)

#### 🎯 포트폴리오 경쟁력

**현재 구현 기준**

* 현재는 Bootstrap 위주만 구현된 상태라면,

  * “전자상거래 전체 시스템을 설계해 두었다” 정도의 가치가 크고,
  * 구현 자체의 경쟁력은 아직 제한적이다.
* 이 경우 별점 평가를 하면 과장일 가능성이 크기 때문에, **현재 상태에 대한 별점은 붙이지 않는 게 맞다.**

**각 Checkpoint 완료 시 예상 경쟁력**

* **Core(Stage 1–4) 완료 기준 – 신입: ⭐⭐⭐⭐ / 이직: ⭐⭐⭐⭐(주니어)**

  * 전자상거래 도메인(상품, 주문, 결제, 재고 등)을 한 번 쭉 구현했다는 점에서 의미 있음.
* **Scale(Stage 5)까지 – 이직(미들): ⭐⭐⭐⭐⭐**

  * Kafka, Elasticsearch, Redis까지 들어가면 “규모/성능/검색”까지 건드린다.
* **Cloud(Stage 6–9)까지 – 이직(시니어): ⭐⭐⭐⭐⭐**

  * 마이크로서비스/클라우드 네이티브까지 들어가면 Tier 1 시니어 포지션 면접에서 충분히 이야깃거리가 된다.

#### 💡 추천 대상

* 전자상거래 도메인 백엔드로 길게 경력 쌓을 계획이 있는 경우
* 긴 기간(6개월 이상)을 프로젝트 하나에 써도 괜찮은 상황

#### 🎯 지원 전략

* **현재 Core 미완성 상태**라면

  * 설계 내용과 지금까지 구현한 부분을 솔직하게 설명하고,
    “입사 후 이쪽 도메인에 기여하고 싶어서 설계까지 먼저 잡아 두었다” 정도로 정리하는 게 낫다.
* **Core 이상이 실제로 구현되어 있을 때**

  * Tier 2(마켓컬리, 무신사 등) 백엔드 포지션에서는 충분히 “실제 비즈니스 문제를 다뤄 본 사람”으로 보일 수 있다.
* **Scale/Cloud까지 구현되면**

  * Tier 1(Coupang, 토스페이먼츠 등)에 “전자상거래/결제 도메인 경력 후보”에 가까운 포지션으로 접근 가능하다.

### 5. backend/node.js (NestJS 패턴 훈련)

#### 📋 프로젝트 개요

* NestJS 10.x, TypeScript, Prisma
* JWT 인증, RBAC, Kafka, Elasticsearch 등

#### 👨‍💻 필요한 개발자 수준

* Node.js/TypeScript 기초가 있는 주니어~미들 정도

#### ⏱️ 구현 기간(현재 기준)

* N2.0–N2.6 합산 5–7주
* 현재 N2.0–N2.6까지 완료된 상태

#### 🎯 포트폴리오 경쟁력 (현재 구현 기준)

* **신입 기준: ⭐⭐⭐**

  * 시장에 많은 “NestJS + RDB + 캐시/검색” 계열 포트폴리오와 비슷한 층.
* **이직 기준: ⭐⭐⭐**

  * Node.js 백엔드 주니어~미들 포지션에 지원할 때 기본기는 충분히 보여준다.
  * “이걸로만 Tier 1을 설득하겠다”기보다는 다른 프로젝트와 조합해서 쓰는 게 현실적이다.

#### 💡 추천 대상

* Node.js 백엔드 포지션을 고려하는 경우
* 빠르게 하나의 백엔드 포트폴리오가 필요할 때

#### 🎯 지원 전략

* backend/node.js 단독으로는 **Tier 3 스타트업** 위주로 보는 게 현실적이다.
* Tier 1–2 쪽을 노리려면 **framework와 묶어서**

  * “프레임워크 내부 구현 경험 + NestJS 실무 패턴 적용” 구조로 가져가는 편이 설득력이 있다.

---

### 6. backend/spring (Spring 패턴 훈련)

#### 📋 프로젝트 개요

* Spring Boot 3.x, Java 17
* JPA, PostgreSQL, Redis, Kafka, Elasticsearch 등

#### 👨‍💻 필요한 개발자 수준

* Java 기초 + Spring Boot 입문 수준의 주니어~미들

#### ⏱️ 구현 기간(설계 기준)

* Milestone 2.0–2.6 합산 5–7주

#### 🎯 포트폴리오 경쟁력

**현재 구현 기준**

* 2025-11 시점 기준으로 **설계만 있고 코드가 없다면**,

  * 이 프로젝트는 아직 “계획” 단계다.
  * 이 상태에서 별점 평가를 붙이면 오해의 여지가 크다.

**Milestone 2.0–2.6까지 실제 구현 완료 시 예상**

* **신입 기준(예상): ⭐⭐⭐**

  * 전형적인 Spring Boot + JPA + RDB 조합.
* **이직 기준(예상): ⭐⭐⭐**

  * Spring 주니어~미들 포지션 지원에 필요한 기본기는 보여준다.
  * “흔하지만 필요한 것”이라는 포지션.

#### 💡 추천 대상

* Spring 백엔드를 아예 포트폴리오에 넣고 싶은 경우
* framework 이후 “실제 프레임워크(Spring)에 적용해 본 사례”로 사용하기

#### 🎯 지원 전략

* **backend/spring 단독**: Tier 3 스타트업 주니어 포지션 정도가 현실적 목표.
* **framework + backend/spring 조합**:

  * Tier 1–2 백엔드 포지션에서 “프레임워크 이해 + Spring 적용”이라는 메시지를 줄 수 있다.

### 7. framework (mini-spring – 프레임워크를 처음부터)

#### 📋 프로젝트 개요

* 원시 소켓 기반 HTTP/1.1 서버
* 라우팅, 미들웨어, DI 컨테이너, 연결 풀

#### 👨‍💻 필요한 개발자 수준

* 소켓/HTTP 프로토콜 이해
* 동시성/스레드 안전성 개념
  → 시니어급이 아니더라도 “시니어가 하는 일에 관심 있는 미들” 정도의 레벨

#### ⏱️ 구현 기간(현재 기준)

* Milestone 1.1–1.4 합산 3–4주
* 현재 Phase 1까지 완료된 상태라고 보는 설계

#### 🎯 포트폴리오 경쟁력 (현재 구현 기준)

* **신입 기준: ⭐⭐⭐⭐**

  * 대다수 신입은 프레임워크를 “쓰는” 쪽만 해 봤기 때문에, 이 프로젝트는 눈에 띈다.
* **이직 기준: ⭐⭐⭐⭐**

  * “Spring/Django 내부가 어떻게 돌아가는지 설명해보라”는 질문에 답할 수 있는 근거가 된다.

#### 💡 추천 대상

* 백엔드 지망자 전반
* “단순 CRUD 개발자” 이미지를 피하고 싶은 사람

#### 🎯 지원 전략

* 대부분의 백엔드 포지션에서 **보조 포트폴리오**로 강력하다.
* 다른 실전 프로젝트와 조합했을 때 효과가 극대화된다.

**현재 구현 기준**

**Milestone 1–6까지 구현 완료 시 예상**

* **신입 기준(예상): ⭐⭐⭐**

* **이직 기준(예상): ⭐⭐⭐**

#### 💡 추천 대상

* 프론트엔드 포지션을 노리는 경우

#### 🎯 지원 전략

* react 단독으로는 Tier 2–3 프론트엔드 주니어용 정도로 보는 게 맞다.
* backend 프로젝트와 함께 가져가서 “풀스택” 스토리를 만드는 용도로 보는 게 현실적이다.

### 2. 학습/구현 시간에 대한 현실적인 보정

* 위 기간은 **“이미 해당 기술을 어느 정도 다뤄본 사람” 기준 구현 시간**에 가깝다.
* 처음 접하는 기술이 많다면 **1.5–2배**까지 늘어나는 것을 전제로 잡는 게 안전하다.
* 회사 준비까지 포함하면, 코딩 외에 **CS/알고리즘/면접 준비 시간**도 별도로 필요하다.

### 3. 이 문서의 사용 방식

* 이 문서는 **“이 포트폴리오가 어느 정도 레벨인지”를 가늠하기 위한 참고 자료**다.
* 실제 채용 결과는

  * 이력서/자소서 완성도
  * CS/알고리즘/시스템 설계
  * 회사 상황
    등에 크게 영향을 받는다.

따라서 이 문서의 별점은 **“현실적인 기대 수준을 맞추기 위한 상대적인 기준”** 정도로 보는 게 맞다.
